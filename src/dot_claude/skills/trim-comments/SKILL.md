# /trim-comments - 冗長コメント削減

対象範囲のコードコメントをプロジェクトのコメント規約に照らして精査し、
冗長なものを削除・圧縮する。

## 使用方法

```
/trim-comments [standard|deep] [--report] [<target>]
```

引数の並びは code-review スキルに準拠: 第1トークンがレベル、`--report` はフラグ、残りが `<target>`。

### レベル（省略時: `standard`）

- **`standard`**: what の言い換え・自明な帰結・重複説明を削除し、長い why コメントは
  要点のみに圧縮する。非自明な why・制約は保持する。
- **`deep`**: 加えて rationale コメントも terse な 1 行に圧縮し、コードと名前から
  推測できるものは削除する。バグ再発防止情報など load-bearing な制約のみ
  最小限の文言で残す。

### フラグ

- **`--report`**: 編集を適用せず、所見の報告（Step 6 の形式）のみ行う。
  デフォルトは適用まで行う。

### `<target>`（省略時: 現在のブランチの変更）

- **commit SHA**: そのコミットが追加・変更したコメントを対象にする（`git show <sha>`）
- **ref range**（`A...B` / `A..B` / `HEAD~n`）: その範囲の diff
- **PR 番号 / GitHub PR URL**: `gh pr diff` で取得した diff
- **ファイル / ディレクトリパス**: そのファイル群に含まれる**全**コメント
- **自由記述**: スコープ制限の指示として解釈し、現在のブランチ diff を起点に絞り込む

## 動作フロー

### Step 1: 対象 diff の決定

target 無指定時は `git diff @{upstream}...HEAD`（upstream がなければ
`git diff main...HEAD`、それも不可なら `git diff HEAD~1`）を取得し、
未コミット変更があれば `git diff HEAD` も範囲に含める。
diff が空なら対象をユーザーに確認する。

### Step 2: コメント規約の読み込み

リポジトリの規約を確認する（この順で探す）:

1. リポジトリルートおよび対象ファイルの祖先ディレクトリの CLAUDE.md
2. コーディング規約文書（例: `docs/coding_style.md` の Comments 節）

規約が見つからない場合は一般原則（why のみ書く・what の言い換えは削除・
周囲のコメント密度に合わせる）を適用する。

### Step 3: 対象コメントの抽出

diff が追加・変更したコメント（target がパスの場合はファイル全体のコメント）を
列挙する。diff 外の既存コメントは編集しない。ただし今回の変更によって
冗長化した近接コメントは所見として報告してよい。

### Step 4: subagent レビュー

対象が数ファイル以上ある場合、または独立レビューが有益な場合は、
subagent に規約とレベルを渡して精査させ、自分の分析と突き合わせる。
観点を分けると効果的（例: what 言い換えの検出 / 複数箇所での同一事実の
重複説明 / 周囲のコメント密度との比較）。

subagent には**保守的になりすぎない**よう明示する。このスキルの目的は
「規約違反の指摘」ではなく「削減」であり、保守的なレビューは
削減候補を取りこぼす。

### Step 5: 判定と適用

各コメントを以下に分類する。判断に迷うものはレベルに従う
（`standard` は保持側、`deep` は削除側に倒す）。

**削除**:

- コードの言い換え（what）: 変数名・関数名・プロパティを読めば分かる説明
- 直後のコード列の逐次説明（レイアウト・幾何・処理手順のナレーション）
- 上位の設定・規約から自明に導かれる帰結
- 同一関数・同一値への参照の反復
- 型注釈を言い換えるだけの `@param` / `@returns`

**圧縮**（複数行 → 要点のみ）:

- 長い why コメント: 非自明な理由・制約の 1〜2 行だけ残す
- `deep` では rationale も terse な 1 行に圧縮する

**統合**:

- 同一の事実が複数箇所で再説明されている場合、最も load-bearing な
  1 箇所にだけ残し、他は削除する

**保持**:

- 非自明な why・制約（外部要因、バグ再発防止情報、コードから読めない前提）
- 公開 API の docstring / JSDoc など、リポジトリの慣例が要求するもの
- コメントアウトされたコードの理由コメント（逆に、理由のない
  コメントアウトコードは所見として報告する）

`--report` 指定時はここで編集せず Step 6 の報告に進む。

### Step 6: 検証と報告

コメントのみの変更はビルド・テスト不要。formatter / lint のみ実行する
（リポジトリに検証用スキルやスクリプトがあればそれを使う。v3 では
`verify` スキル）。コミットは指示があるまで行わない。

報告は箇所ごとの表にまとめる:

| 箇所 | 変更 | 内容 |
|---|---|---|
| `file.ts` `funcName` | 4行→1行 | 幾何の逐次説明を削除、中央基準の理由のみ残す |
| `file.css` `.selector` | 削除 | `width:${value}%` の言い換え |

表の後に「削った内容の種別」と「保持したコメントとその理由」を簡潔に述べる。
保持理由は具体的に（例: unmount で null 化されない制約はバグ再発防止情報）。
